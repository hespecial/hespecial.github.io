使用 Redis 实现分布式锁

原理

  • 核心是利用 Redis 的 SETNX 命令的原子性
  • 当多个客户端同时执行 SETNX 命令时，只有一个客户端能成功设置锁，其他客户端会失败

加锁

  • 使用 SET key value NX EX 10 命令
    • key 是锁的名称
    • value 是锁的唯一标识，一般使用 uuid，保证释放的是自己的锁
    • NX 表示仅当 key 不存在时才设置，防止覆盖
    • EX 设置锁的过期时间，防止死锁

释放锁

  • 因为要保证释放的是自己的锁，所以要先执行 GET 进行校验，校验成功后使用 DEL 删除 key
  • 一般使用 Lua 脚本执行，保证 GET 和 DEL 是原子操作

使用 Zookeeper 实现分布式锁

原理

  • 核心是利用 Zookeeper 的临时顺序节点的特性和 Watch 机制
  • 每个临时顺序节点就是 Zookeeper 中某个目录下的一个文件
    • 临时：客户端断开连接后，节点会自动删除，可以防止死锁
    • 顺序：节点创建时对应文件会自动加一个自动递增的编号，可以配合watch监听上一个节点的状态

加锁

  1. 使用Create命令在指定目录创建一个临时顺序节点
  2. 获取目录下所有的子节点并排序
  3. 判断自己是否是最小的节点
    • 如果是，说明成功获取了锁
    • 如果不是，说明锁被其他客户端获得，接下来监听前一个节点的删除事件，事件触发后，当前客户端就能获取到锁了

释放锁

  • 直接删除当前节点即可

使用 ETCD 实现分布式锁

原理

  • 核心是通过原子操作(Compare-And-Swap，CAS)和ETCD的租约机制(lease)来保证锁的唯一性和自动释放
  • 当客户端持有锁时，如果进程崩溃或网络断开，租约过期后锁会自动释放，避免死锁
  • 通过 Watch 机制 监听锁的变化，可以实现等待锁的功能

加锁

  1. 使用Lease机制创建一个带生存时间的会话，如果客户端崩溃，锁会自动释放
  2. 尝试加锁时，指定锁的名称，然后使用事务CAS保证原子性
  3. 如果业务执行时间可能超过租约TTL，可以仿照看门狗机制，定期续约租约，防止锁过期

释放锁

  • 删除key并撤销租约即可
